1 What this is & how to use it

The starter repo already ships with three working modules — Customer, Article and Order — all following the same layering pattern. Your job is to add a fourth module, Supplier, but without writing Java first: you write a short English brief, then let Spec Kit turn it into a spec, a plan and a task list that you review, push back on, and then implement.

The goal is not to finish a Supplier module. The goal is to feel the rhythm of spec-first development and develop an instinct for what each slash command is actually good for.

How the hints work

Throughout the lab you'll see expandable hint boxes. They contain coaching — what a step is for, what to watch out for, what "good" looks like. Don't open them on the first pass. Try the step yourself, form your own opinion, then peek.

2 Setup — verify the starter works

You need Java 17 and Maven on your PATH. From the repo root:

mvn test
mvn package
java -jar target/erp-starter.jar

You should see 13 tests pass and a CLI menu with Customers / Articles / Orders / Quit. Poke around — list customers (1 → 1), print the CSV report (1 → 4), quit with q.

If this fails

Fix setup before moving on. The whole point of a brownfield workshop is that the existing tests stay green after your changes.

4 The feature brief

Our purchasing team needs a Supplier maintenance inside the ERP. A supplier has a code, legal name, country (ISO-3166 alpha-2), VAT number, default payment term in days and an Active flag. Each supplier may have zero or more contacts (name, role, email, phone). The system must refuse to save a supplier with a VAT number that is invalid for the chosen country, and refuse to deactivate a supplier that still has open orders (any order not in status CLOSED). Purchasing managers need a printable "Active suppliers" list sorted by country, then by legal name. The new module must plug into the existing CLI as option 4, next to Customers, Articles and Orders. As a stretch goal, we would like a small local HTTP endpoint returning the active suppliers as JSON so the portal team can prototype against it.

Deliberately a little vague — /speckit.clarify has room to do real work.

5 Run /speckit.constitution Constitution

Before specifying anything, tell Spec Kit the ground rules every future step must respect. The constitution is the short list of non-negotiables for this codebase — style, architecture, testing, dependencies — that later commands (specify, plan, implement) will treat as guardrails.

/speckit.constitution
Tip — constitution captures what never changes

Think of it as a tiny project charter. Good clauses are durable ("no new runtime dependencies without approval", "every service has a JUnit test", "mirror the layering of the existing modules") — not feature-specific. You write it once per project and amend it rarely.

You'll feel its value two steps later: /speckit.plan will stop suggesting frameworks you don't want, because they violate your own rules.

Guardrail — keep it short and enforceable

A constitution with 40 clauses gets ignored. Aim for 5–10 crisp, testable statements. If you can't tell whether a future diff violates a clause, the clause is too vague — rewrite it until you can.

Peek — principles that fit this exercise
  • Pure JDK only — no new Maven dependencies beyond JUnit.
  • In-memory persistence — no database, no file I/O for state.
  • Follow the existing layering — every new module mirrors com.example.erp.customer (model, repository, rules, service with beforeSave / beforeDeactivate hooks, report, CLI).
  • Tests first, tests always green — every service has JUnit tests; mvn test stays green after every task.
  • Business rules raise exceptions, not booleans — invalid state fails loudly with a clear message.
  • CLI is the user-facing surface — every module plugs into ErpApp's menu.

6 Run /speckit.specify Specify

In your GitHub Copilot CLI / Chat, invoke:

/speckit.specify

and feed it the brief from §4. Read the generated spec carefully and challenge anything that doesn't match what you actually asked for.

Tip — specify is about the what and why, not the how

A good brief (and the spec that comes out of it) describes the outcome the business wants and why they want it: who the users are, what problem is being solved, what has to be true for the feature to count as "done". It should not yet talk about files, packages, classes, HTTP routes or algorithms — that's the plan's job.

When you review the output, ask yourself: "Could a backend dev, a frontend dev and a tester all read this and agree on what's being built, without me explaining anything?" If yes, the spec is doing its job.

Guardrail — don't let the spec leak implementation

If specify comes back already naming Java classes, database tables, or specific libraries, that's a smell. Prune those out and keep the spec at the level of observable behaviour: "the system refuses to save when VAT is invalid" — not "SupplierService.beforeSave throws RuleException". Decisions like the latter belong in the plan, where they can be challenged independently.

7 Run /speckit.clarify Clarify

/speckit.clarify

This step matters more than any other. Expect questions around VAT validation, the open-orders check, how Active is modelled, the report format, and the stretch HTTP endpoint.

Tip — clarify is your quality gate

It's the Spec Kit step most teams skip — and it's the one that usually catches a missing requirement before it becomes a bug. Run it every time, and actually read the questions. Each one is the tool telling you it saw an ambiguity; dismissing the question doesn't resolve the ambiguity, it just hides it until implementation.

Answer with the smallest workable choice. "Refuse with a clear error message" beats "design an error-code taxonomy". The time to expand scope is later, with evidence.

Guardrail — don't let clarify invent new requirements

If a clarifying question proposes something that isn't in the brief (audit fields, soft delete, a second report, a DELETE endpoint), push back. Narrow the question, or answer "out of scope for this iteration". A clean spec is one where every bullet is traceable to the brief or to an answer you explicitly gave.

Peek — the ambiguity to watch for in this exercise

The "open orders" rule is a trap. The starter's Order has a customerCode field, not a supplierCode. clarify should surface this. When it does, don't paper over it — pick one approach explicitly and record the choice in the spec (e.g. "add a separate SupplierOrderRepository", or "inject an OpenOrdersGuard interface"). Either is fine; ambiguity is not.

8 Run /speckit.plan Plan

/speckit.plan

With the clarified spec in hand, the plan should turn "what" into "how" — and it should sound like it was written by someone who has actually read this repo.

Tip — plan is about the how

If specify answers "what and why", the plan answers "how, concretely". A good plan names real files, lists the interfaces, calls out the test strategy, and orders the work so each step leaves the build green. If the plan is still at the level of "add a service layer for suppliers", it's too abstract — push it until it says "add SupplierService.java with beforeSave(Supplier), mirroring CustomerService".

Guardrail — reject plans that drift from the existing pattern

The pom.xml has only JUnit. If the plan suggests Jackson, Spring, Lombok, an ORM, or any other dependency, that's a signal to re-prompt more constrained: "stay on pure JDK, no new Maven dependencies, mirror the layering of com.example.erp.customer". Consistency with the existing code is worth more than any shiny library.

Peek — shape of a "good" plan for this exercise
  1. New package com.example.erp.supplier: Supplier, SupplierContact, SupplierRepository, InMemorySupplierRepository, SupplierService, SupplierListReport, SupplierCli.
  2. Reuse VatValidator (move to a shared common.rules package if it helps, or keep it where it is and import it — document the choice).
  3. Open-orders rule: implement the option chosen in §6.
  4. Wire into ErpApp as menu option 4; extend DataSeeder with 2–3 sample suppliers.
  5. Tests mirroring CustomerServiceTest: happy save, invalid VAT, blank name, negative term, deactivate-allowed, deactivate-refused.
  6. Stretch: SupplierHttpServer using com.sun.net.httpserver.HttpServer, GET /suppliers returns JSON.

9 Run /speckit.tasks Tasks

/speckit.tasks
Tip — tasks is about slicing

A task is well-sliced when it maps to one commit, leaves the build green, and has an obvious "done" signal (a test passes, the CLI prints the new option, etc.). Where possible, order tests before the code that makes them pass, so each pair of tasks is a tiny red/green cycle. That way, at any point you can stop and hand off without leaving a half-finished branch.

Peek — expected task shape for this exercise
  1. Create com.example.erp.supplier package with Supplier + SupplierContact.
  2. Add SupplierRepository + InMemorySupplierRepository.
  3. Decide where VatValidator lives (shared package vs. import in place).
  4. Add SupplierService with beforeSave — tests first, watch them go red.
  5. Make the beforeSave tests green.
  6. Add the open-orders rule wiring chosen in §6; beforeDeactivate tests red → green.
  7. Add SupplierListReport + a small test.
  8. Add SupplierCli; wire menu option 4 in ErpApp.
  9. Extend DataSeeder with sample suppliers.
  10. mvn verify + manual smoke test from the CLI.
  11. (Stretch) Add SupplierHttpServer, hit it with curl http://localhost:8080/suppliers.

10 Run /speckit.implement Implement

Walk through tasks 1–5 together in the session. Tasks 6–11 are a take-home.

/speckit.implement
Tip — implement is where you stay in the loop

Run mvn test after every task. When a test goes red, decide — before asking the tool to fix anything — whether it's the test that's wrong, the implementation that's wrong, or the spec itself that's wrong. That triage is the skill the workshop is training. If you skip it, you're back to vibe-coding with extra steps.

Guardrail — commit after every task

A per-task commit history is the easiest way to back out if the tool drifts on task n+1, and it's the cleanest artefact to show a reviewer afterwards — the Git log becomes a literal walk-through of the specified work.

11 Definition of done

  • mvn test stays green and adds at least the six Supplier tests.
  • From the CLI, option 4 opens a Supplier maintenance that can create, list, deactivate and report.
  • Saving a supplier with an invalid VAT is refused with a clear message.
  • Deactivating a supplier with at least one non-CLOSED order is refused.
  • The "active suppliers" CSV report prints sorted by country, then legal name.
  • (Stretch) curl http://localhost:8080/suppliers returns JSON listing the active suppliers.
  • The jar still runs with java -jar target/erp-starter.jar.

12 What you've just practised

Six distinct skills — each matched to a slash command — that travel with you to any codebase or language:

StepSkillQuestion it answers
/speckit.constitutionSetting guardrailsWhat rules does every change in this codebase have to respect?
/speckit.specifyStating the problemWhat do we need, for whom, and why?
/speckit.clarifySurfacing ambiguity earlyWhat questions do I owe an answer to before I start?
/speckit.planDesigning the solutionHow will this land in this codebase?
/speckit.tasksSlicing the workWhat's the smallest shippable step — then the next?
/speckit.implementExecuting with disciplineIs the test, the code, or the spec wrong when something breaks?

The loop is the point. Keeping spec, plan, tasks and code in sync is what turns AI-assisted coding into AI-assisted engineering — and it works on any stack you bring it to.
