1 What this is & how to use it

Lab 1 teaches spec-first development with Spec Kit. This lab is the companion exercise: before asking an agent to implement a feature, make the repository easier for agents to understand.

The goal is not to chase a perfect score. The goal is to learn how to read an AI-readiness report, identify which missing context actually matters for this repo, and generate useful agent instructions without accepting generic boilerplate.

Lab posture

Treat every AgentRC output as a draft. AgentRC reads the repository and proposes context, but you are still responsible for checking whether it matches the real Java/Maven/JUnit stack and the workshop constraints.

2 Setup - verify tools and repo state

Run from the repository root. You need Node.js with npx, Git, and the usual Java/Maven setup from Lab 1.

git status --short
node --version
npx --version
mvn test

Start from a clean or intentionally understood worktree. If you already have generated instruction files or custom agents under .github/, note that before running AgentRC so you can tell what changed.

Tip - capture the baseline first

Run git status --short before and after each AgentRC command. The diff is the easiest way to separate "AgentRC generated this" from "this was already in my repo".

Guardrail - don't skip the build

AI-readiness is not only documentation. A coding agent also needs reliable commands that prove it did not break the project. For this repo, mvn test is the core validation signal.

3 Run agentrc readiness Assess

Ask AgentRC to score the repository's current support for AI-assisted development:

npx github:microsoft/agentrc readiness

Read the report before changing anything. You are looking for the difference between "the app works for a human" and "the repo gives a coding agent enough context to work safely".

Tip - read across pillars, not just the headline score

A low score in one pillar can be more important than a high overall average. For example, missing test commands or missing repo instructions will hurt agent reliability more than a missing changelog in this small workshop repo.

Peek - what AgentRC is likely to notice here

This repo has a clear README, a Maven build, JUnit tests, and a GitHub Pages workflow. It may still lack formal linting, a dev container, MCP server configuration, issue/PR templates, dedicated evals, and complete agent-facing instructions.

4 Interpret the 9 pillars and 5 levels

AgentRC frames readiness as a maturity model. Use it to decide where better context will most reduce agent mistakes.

DimensionWhat to look for in this repo
DocumentationDoes the README explain purpose, commands, and module shape well enough for an agent?
Build systemAre Maven commands obvious, reliable, and documented?
TestingAre JUnit tests present, named clearly, and easy to run?
Code styleAre formatting and layering conventions explicit, or only implied by examples?
SecurityAre secret-handling and dependency expectations clear, even for a tiny starter?
EnvironmentCan a fresh machine understand required tools without tribal knowledge?
Task discoveryAre issues, PRs, or workshop instructions structured enough to guide work?
Agent contextAre Copilot instructions, MCP configs, and generated guidance present and accurate?
EvaluationIs there any way to evaluate whether generated agent guidance is good?
Important

The exact pillar names may vary as AgentRC evolves. Keep the lab focused on the habit: translate each score into a concrete missing piece of context that would help an agent operate safely in this repo.

5 Find the missing context Diagnose

Turn the readiness report into a short action list. For each finding, ask: "Would this help a coding agent make better changes here?"

  1. Mark findings that block safe code changes, such as missing test commands or missing module conventions.
  2. Mark findings that improve onboarding, such as dev environment notes or README gaps.
  3. Mark findings that are nice-to-have for this workshop, such as full production observability.
  4. Ignore or defer suggestions that would distort the repo, such as adding a framework just to satisfy a checklist.
Guardrail - readiness is context, not ceremony

This is a tiny plain-Java starter. Do not add heavyweight tooling just to raise a maturity score. The best improvement is the smallest piece of context that makes future agent work more correct.

Peek - examples of useful improvements
  • Document that new modules mirror com.example.erp.customer.
  • State that mvn test must stay green after every change.
  • Explain that services use beforeSave and beforeDeactivate hooks.
  • Record that the app intentionally has no database, no framework, and in-memory repositories.
  • Generate evals that check whether an agent preserves those constraints.

6 Run agentrc instructions Generate

Now ask AgentRC to generate tailored agent context from the actual repository:

npx github:microsoft/agentrc instructions

The key promise: no static templates. AgentRC inspects this Java/Maven/JUnit codebase and produces instruction files, evals, and developer configs that should reflect what is really here.

Tip - watch the file list

After the command finishes, run git status --short. New or modified files are the workshop material for the next step. Do not commit them until you have read them.

Guardrail - generated does not mean approved

AgentRC should infer the stack, but you still review for hallucinated tools, wrong commands, or guidance that conflicts with the existing project. Reject instructions that suggest Spring, a database, Lombok, or unrelated build tools for this starter.

7 Review what AgentRC generated Review

Open each generated file and check it against the codebase. This is where the lab turns from "run a command" into engineering judgement.

CheckQuestion
Stack accuracyDoes it identify plain Java, Maven, JUnit 5, and no framework?
CommandsDoes it name the real validation commands: mvn test, mvn package, and the jar command?
ArchitectureDoes it describe model/repository/service/rules/CLI layering accurately?
ConstraintsDoes it preserve no database, in-memory repos, and no new runtime dependencies without approval?
EvalsDo generated evals measure useful behaviour for this repo rather than generic style?
MCP/dev configAre any MCP server or dev environment configs relevant, safe, and documented?
Peek - a good generated instruction for this repo

A strong instruction says something like: "When adding a module, mirror customer; add JUnit tests for rule failures; keep repositories in memory; run mvn test; do not add Spring or persistence." That is repo-specific context an agent can act on.

8 Decide what to keep Curate

Make a deliberate keep/edit/drop decision for every generated artifact. The output is not valuable because a tool created it; it is valuable only if it makes the next AI-assisted change safer.

git diff
mvn test
git status --short

If the generated context is accurate, commit it separately from feature work. If it needs correction, edit it until it reflects the actual repo.

Suggested commit shape

Keep AgentRC-generated repo context in its own commit. That makes it easy for reviewers to inspect the new instructions before any later AI-assisted implementation depends on them.

9 Definition of done

  • You ran npx github:microsoft/agentrc readiness from the repo root.
  • You can explain the repo's strongest and weakest readiness pillars.
  • You identified which missing context matters for this Java/Maven starter and which suggestions are out of scope.
  • You ran npx github:microsoft/agentrc instructions.
  • You reviewed generated instructions, evals, and dev configs before accepting them.
  • mvn test still passes after any kept changes.
  • The final diff contains only context you understand and are willing to maintain.

10 What you've just practised

This lab gives you a repeatable AI-readiness loop for any brownfield repository:

StepSkillQuestion it answers
readinessAssessmentWhat context does this repo already give an agent?
Score reviewPrioritisationWhich missing context will reduce real mistakes?
instructionsGenerationWhat repo-specific guidance can be created from the code itself?
Human reviewCurationIs the generated context true, useful, and safe to keep?
CommitOperationalisationCan future agents and teammates rely on this context?

The loop is the point. Readiness is not a badge; it is the practice of giving agents the same context a careful teammate would need before touching the code.
